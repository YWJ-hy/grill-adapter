## grill-adapter conventions (plain Claude Code host)

Generated by grill-adapter. This block wires grill-adapter's standalone skills, hooks, and engine into a plain Claude Code workflow **by convention only ‚ÄĒ no host skill is patched**. `__GRILL_ADAPTER_ROOT__` is the installed grill-adapter payload root. There is no fixed planning framework here, so you invoke each touchpoint yourself at the matching moment.

### Intake ‚ÄĒ Lanhu (only when a Lanhu link is given)

If the user gives a Lanhu URL/invite link, run `/lanhu-requirements <link> frontend|backend <optional name>` first, confirm the `.lanhu/.../index.md` evidence package, and use it as requirements input only ‚ÄĒ never copy Lanhu content into wiki, spec, plan, acceptance, or tests.

### Disclose ‚ÄĒ before proposing an approach

Run `/wiki-research` (phase `brainstorm`) to surface relevant project/shared wiki as lightweight context. If none applies, record a one-line N/A and continue. Write no sidecar here.

### Verify ‚ÄĒ while drafting a spec/plan

Run `/source-truth-check` to render the source-of-truth policy (`spec-pre` for a spec, `plan-pre` / `plan-review` for a plan) and honor any block: do not edit `truth/edit: never` paths; require explicit authorization for `truth/edit: ask` paths.

### Carry ‚ÄĒ when writing an implementation plan

Run `/wiki-research` (phase `plan`), then follow it to scaffold and finalize the `<plan-stem>.wiki-context.json` sidecar (`wiki_context_render.py --scaffold` ‚Üí edit each section's `destination` once ‚Üí `--finalize`). Link the sidecar from a `## Referenced Project Wiki` section and route each section to the numeric plan task ids in `destination.tasks`.

gitignore-aware: before committing a plan/spec + `.wiki-context.json`, run `git check-ignore -q <path>`; if ignored, do not commit and never `git add -f`. `.wiki-candidates.jsonl` is transient scratch, never committed.

### Bind ‚ÄĒ before implementing each task

Run `/wiki-materialize <task-id>` to reread that task's authoritative hard-constraint sections (local + `github_mcp`, bounded 1-hop `depends-on` closure); run `--fingerprint-preflight` once before the first task. The `wiki-reread` hook is a coarse session-level backstop; the per-task call is the precise path. The `source-truth-lint` hook lints real changed files; resolve any `block`/`ask` before completing the task.

While implementing, append durable decisions/gotchas as JSONL lines to `<plan-stem>.wiki-candidates.jsonl` and keep going.

### Capture ‚ÄĒ after the work is reviewed/accepted

1. If using grill-style knowledge files, convert the increment: `python3 __GRILL_ADAPTER_ROOT__/scripts/grill_context_to_candidates.py <repo-root> --since <branch-point> --out <plan-stem>.wiki-candidates.jsonl` (bridge, not `import-wiki`).
2. Run `/update-wiki`, consuming the candidates sidecar, to make the keep-or-skip determination. The `wiki-capture` hook (Stop) reminds you when candidates are still pending.

### Debug

After root-cause evidence narrows the failure, optionally run `/wiki-research` (phase `debug`, ‚Č§2 sections) for a targeted lookup. After a verified fix, run `/break-loop` for a retrospective; it hands durable candidates to `/update-wiki`.

### Boundary

grill-adapter's standalone utility skills are utilities, not workflow steps; running one does not imply any implementation/review is complete or verified.
