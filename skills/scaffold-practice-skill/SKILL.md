Generated by grill-adapter.

Use this skill to turn a reusable engineering practice into a **layered skill pack**, or to restructure an existing skill into that shape, then stage one atomic **Obsidian Skill Card candidate** so reviewed Capture can publish it.

This is an adapter authoring skill. Its completion proves only that a versioned pack was validated and a registration candidate is pending. It is not discoverable until Capture applies it, publishing creates a reviewed PR, a human merges that PR, and the configured base worktree synchronizes.

## When to use

- A reusable practice keeps getting re-explained (management-page layout, micro-app host/child file structure, form/table conventions, a fixed review checklist) and should become an executable skill.
- An existing monolithic `SKILL.md` is large and loads in full for both implement and review — restructure it so heavy content loads on demand.
- `update-wiki` handed off a **skill-pack candidate** (reusable workflow/process knowledge that belongs in an executable skill, not a wiki leaf page).

## Pack model (one fixed file, open beyond that)

The only invariant is that **`SKILL.md` is a thin router**: frontmatter (`name` + a trigger-rich `description`), when-to-use, mode selection, a few core hard rules, and pointers to sibling files. Everything else is an **open set** created only when the content warrants it:

```
.claude/skills/<name>/
  SKILL.md          # required: thin router
  implement.md      # optional (common): implementation flow
  review.md         # optional (common): review checklist
  rules.md          # optional (common): full normative rules
  examples.md       # optional (common): examples
  scripts/  references/  templates/  assets/  ...   # any other files/dirs
```

**Why only SKILL.md is fixed**: progressive disclosure — keep SKILL.md small so an implement task pulls implement content and a review task pulls review content, instead of loading everything every time; a short `description` also matches more reliably. The full content still lives in the pack, so the skill works outside any specific host.

Do not invent a fixed five-file template. Split by what the content actually needs (a pure generator may have no `review.md`; a small skill stays small).

## Framework-independent authoring

Author each pack so it works with or without a specific host workflow. In `SKILL.md`, prefer this context priority rather than hard-coding any wiki path:

1. project rules already present in the current context;
2. a wiki/doc/spec path the user provides;
3. locally discoverable project docs;
4. this skill's generic flow.

If project-specific rules are missing, ask for the relevant practice document before making irreversible design choices. **Do not maintain a "which wiki to read" routing table inside the skill** — the wiki points at the skill, not the other way around.

## Mechanical helper

All pack file operations, contract hashing, candidate staging, and validation are mechanical and belong to the script. You do the semantic work (naming, versioning, the `description`, trigger scenarios, roles, file split, and prose).

```bash
# create a new pack skeleton (open file set; only SKILL.md is mandatory)
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/scaffold_practice_skill.py \
  --project-root . --json scaffold --name <skill-name> \
  --version <semver> --description "<trigger-rich one-liner>" \
  --files implement.md,review.md,rules.md

# convert an existing skill (NON-DESTRUCTIVE): stages a new pack, carries over all
# bundled files (scripts/, assets), and reports source content not yet represented
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/scaffold_practice_skill.py \
  --project-root . --json convert --from <path/to/old/SKILL.md-or-dir> \
  --name <skill-name> --files rules.md,review.md

# validate the pack and stage a content-addressed registration candidate in the
# feature journal. This does not write Obsidian or a discovery index.
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/scaffold_practice_skill.py \
  --project-root . --json stage-card --name <skill-name> \
  --feature-slug <feature-slug> --provider claude-code-project \
  --version <same version as SKILL.md> \
  --triggers "<scenario>,<scenario>" --summary "<one-line theme summary>" \
  [--roles implement,review|review|implement]

# validate pack invariants (+ optional source coverage)
python3 ${CLAUDE_PLUGIN_ROOT}/scripts/scaffold_practice_skill.py \
  --project-root . --json validate --name <skill-name> [--from <old SKILL.md-or-dir>]
```

## Workflow

### Create

1. Confirm the project root and one feature slug for the current workflow.
2. Choose a kebab-case `name`, semantic `version`, trigger-rich `description`, and the sibling files the content needs.
3. Run `scaffold` to write the skeleton, then author the real content into `SKILL.md` and each file. Keep `SKILL.md` thin; push heavy rules/examples into siblings.
4. Run `validate --name <name>` and fix every pack error and router-reference warning before staging.
5. Run `stage-card` with the same version, authored summary, explicit triggers, and matching roles. The helper hashes every pack file, rejects symlinks/version drift, and appends one idempotent `pending` registration to the feature journal.
6. Hand the pending candidate back to `update-wiki`; do not author an Obsidian Note or discovery index here.

### Convert (non-destructive)

1. Run `convert --from <existing skill>`; it stages a new pack, carries over every bundled file (including `scripts/`), and lists `uncovered` source content.
2. Author the layered files so every `uncovered` heading/section is represented somewhere; never drop content. Bundled non-markdown files are already carried over.
3. Run `validate --name <name> --from <existing skill>` until there are no coverage gaps.
4. Show the user the new pack and a diff against the original. **Replace the original only after the user confirms** (the script never deletes the original). If the staged pack was written to `<name>.converted/`, move it into place on confirmation.
5. After the user confirms replacement, validate and run `stage-card` exactly as in Create.

## Publishing boundary

`stage-card` writes only append-only local journal state. `update-wiki` remains the semantic and authorization gate: it renders one atomic `type: guide` Card with `skill_provider`, `skill_name`, `skill_version`, `skill_contract_hash`, `skill_roles`, and `skill_triggers`; then it uses the normal Obsidian proposal/apply receipt and draft-PR publisher. A staged or open-PR Card is `pending`, not discoverable.

## Boundaries

- Do not copy or replace the official `skill-creator`. The user may start a new skill with `skill-creator`; this skill normalizes it into a layered pack and registers the discovery card.
- Do not move normative content into the Card. The atomic Card carries discovery and binding metadata; the pack holds executable content.
- Do not write or regenerate legacy `guides/skills.md`; Skill Card discovery comes only from merged, base-synchronized Obsidian Notes.
- Convert is non-destructive: never delete or overwrite the original skill without explicit user confirmation, and never lose a bundled file.

## Completion checklist

- [ ] `SKILL.md` is a thin router with a trigger-rich `description`; heavy content lives in sibling files.
- [ ] Only the files the content needs were created (open set, not a forced template).
- [ ] For convert: all source headings/sections are represented and all bundled files (incl. `scripts/`) were preserved; the original was replaced only after user confirmation.
- [ ] `SKILL.md` carries the staged version; `validate` passes; `stage-card` reports the provider/version/roles/triggers/contract hash and `discoveryState: pending`.
- [ ] The pack is usable outside any specific host (context-priority phrasing, no hard-coded wiki routing table).
